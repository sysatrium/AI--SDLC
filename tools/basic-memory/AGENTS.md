## 📥 Протокол начала сессии (АВТОМАТИЧЕСКИ)

Выполнять при каждом запуске **без ожидания команды от пользователя**:

```
1. read_note("memory://me/user-profile")             → кто пользователь
2. read_note("memory://me/preferences")              → как работать
3. read_note("memory://project-alpha/facts")         → реальность проекта
4. read_note("memory://project-alpha/decisions")     → принятые решения
5. read_note("memory://project-alpha/constraints")   → что НЕЛЬЗЯ делать сейчас
6. read_note("memory://project-alpha/open-questions") → открытые вопросы
7. read_note("memory://project-alpha/last-session")  → где остановились
```

После загрузки кратко подтвердить:
"Контекст загружен. Продолжаем с [последний шаг из last-session]."

## 📤 Протокол завершения сессии (АВТОМАТИЧЕСКИ)

Выполнять в конце сессии или при словах пользователя "стоп", "заканчиваем", "конец сессии", "всё на сегодня":

```
write_note("memory://project-alpha/last-session") с содержимым:
  - Дата и ветка git
  - ✅ Что завершено
  - 🔄 Что в процессе
  - ⏭️ Следующие шаги (пронумерованный список)
  - ❓ Вопросы, возникшие в сессии
  - ⚠️ Блокеры если есть
```

## 🔄 Правила во время сессии

### При каждом новом архитектурном или значимом решении:
```
edit_note("memory://project-alpha/decisions")
→ Добавить блок ADR-NNN: решение, контекст, обоснование, последствие, компромисс, дата
→ В open-questions.md перенести закрытый вопрос в секцию "Решено"
```

### При обнаружении нового факта о проекте:
```
edit_note("memory://project-alpha/facts")
→ Добавить строку [fact] с соответствующим тегом
```

### При возникновении вопроса без немедленного ответа:
```
edit_note("memory://project-alpha/open-questions")
→ Добавить в нужную секцию приоритета (Критичные / Обсуждаются)
```

### При обнаружении ограничения:
```
edit_note("memory://project-alpha/constraints")
→ Добавить [constraint] с: датой/событием снятия, причиной, тегом
```

### Перед любым предложением изменения кода:
```
read_note("memory://project-alpha/constraints")
→ Убедиться что изменение не нарушает активные ограничения
→ Если нарушает — предупредить пользователя до начала работы
```

### При изменении предпочтений пользователем:
```
edit_note("memory://me/preferences")
→ Обновить соответствующую строку [preference]
```

## 🚫 Жёсткие правила (никогда не нарушать)

- Никогда не хранить пароли, токены и секреты в файлах памяти
- Никогда не предлагать код без TypeScript типов (strict mode)
- Никогда не предлагать платные инструменты без сравнения с бесплатными альтернативами
- Никогда не начинать рефакторинг без явного подтверждения пользователя
- Никогда не игнорировать активные ограничения из constraints.md
- Никогда не начинать реализацию новой фичи без спецификации
