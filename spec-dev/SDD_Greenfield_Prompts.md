# Spec-Driven Development: GREENFIELD Playbook

> **Сценарий:** Новый проект с нуля. Нет легаси кода, нет существующих ограничений архитектуры.
>
> **Цель:** Написать спецификации ДО написания кода, используя их как исполняемые контракты для AI-агентов.

---

## 🧠 Концепция: Что такое SDD

Spec-Driven Development (SDD) — это методология, где **формальные спецификации становятся исполняемыми контрактами**, из которых AI-агенты генерируют код. В отличие от "vibe coding" (разговорные промпты без структуры), SDD обеспечивает:

- Архитектурную согласованность между сессиями
- Трассируемость решений (почему принято именно так)
- Воспроизводимые результаты при регенерации кода
- Встроенные проверочные критерии (что значит "готово")

**Цикл:** `Specify → Plan → Tasks → Implement → Verify`

## 🚀 ФАЗА 0: Инициализация проекта

### Промпт 0.1 — Создание Constitution (Конституция проекта)

```
Я начинаю новый проект: [ОПИСАНИЕ ПРОЕКТА В 2-3 ПРЕДЛОЖЕНИЯХ].

Задай мне подробный опросник для создания constitution.md — основного документа,
который будет управлять всеми решениями AI-агентов в этом проекте.

Опросник должен охватывать:
1. Технологический стек (языки, фреймворки, БД, инфраструктура)
2. Архитектурные принципы (монолит/микросервисы, DDD, CQRS и т.д.)
3. Соглашения по коду (нейминг, структура файлов, паттерны)
4. Запрещённые практики (что AI НЕ должен делать)
5. Внешние зависимости и ограничения
6. Требования к качеству (покрытие тестами, безопасность, производительность)
7. Процесс ревью и деплоя

После моих ответов — сгенерируй .specify/memory/constitution.md
```

### Промпт 0.2 — Генерация AGENTS.md

```
На основе constitution.md создай файл AGENTS.md для корня репозитория.

Этот файл — постоянный контекст для всех AI-агентов в проекте. Он должен содержать:

1. **Project Overview** — одна страница о том, что мы строим и зачем
2. **Tech Stack** — точные версии, без "или/или"
3. **Architecture Decisions** — ключевые ADR в формате: Решение / Причина / Последствия
4. **Code Conventions** — конкретные правила с примерами
5. **Forbidden Patterns** — что никогда не делать (с объяснением почему)
6. **Testing Strategy** — уровни тестирования и минимальный coverage
7. **Security Rules** — обязательные проверки
8. **File Organization** — как называть файлы и где их класть

Формат: Markdown, максимально конкретный, без воды.
Цель: AI-агент должен прочитать этот файл и понять ВСЁ необходимое для работы.
```

## 📋 ФАЗА 1: Спецификация фичи

### Промпт 1.1 — Создание spec.md

```
Создай spec.md для фичи: [НАЗВАНИЕ ФИЧИ]

Контекст проекта: [ССЫЛКА НА AGENTS.md или краткое описание]

Spec должен содержать 6 обязательных элементов:

1. **Outcomes (Результаты)** — конкретные, измеримые результаты:
   "Пользователь может X, система делает Y, метрика Z достигнута"
   НЕ: "реализовать авторизацию"
   ДА: "Пользователь входит через email/пароль, получает JWT-токен (15 мин),
        refresh token (7 дней), сессия сохраняется между перезагрузками"

2. **In-Scope / Out-of-Scope** — явный список что включено и что ИСКЛЮЧЕНО
   (Out-of-scope не менее важен — AI расширяет скоуп если не запрещено)

3. **Constraints & Assumptions** — ограничения стека, API, производительности,
   решения которые уже приняты и не обсуждаются

4. **Decisions Already Made** — что уже решено: схема БД, библиотеки, паттерны

5. **Task Breakdown** — предварительная декомпозиция на 5-10 атомарных задач

6. **Verification Criteria** — acceptance criteria + edge cases + какие тесты должны пройти

Бизнес-контекст: [ЧТО ЭТА ФИЧА ДАЁТ ПОЛЬЗОВАТЕЛЮ/БИЗНЕСУ]
```

---

### Промпт 1.2 — Создание контракта API (OpenAPI)

```
На основе spec.md для фичи [НАЗВАНИЕ] создай OpenAPI 3.0 спецификацию.

Требования:
- Все эндпоинты из spec.md
- Полные схемы запросов и ответов (не $ref-заглушки)
- HTTP коды для success и всех error-случаев
- Примеры (examples) для каждого эндпоинта
- Security схемы согласно constitution.md
- Описание каждого поля (description обязателен)

Дополнительно укажи:
- Какие эндпоинты требуют аутентификации
- Rate limiting требования
- Идемпотентность (особенно для POST/PUT)

Сохрани как: specs/[номер]-[feature-name]/contracts/api-spec.yaml
```

## 🗺️ ФАЗА 2: Планирование

### Промпт 2.1 — Создание plan.md

```
На основе spec.md создай plan.md — архитектурный план реализации.

plan.md должен содержать:

1. **Component Diagram** (текстовый, Mermaid) — что с чем взаимодействует

2. **Technology Choices** — какие конкретно библиотеки/паттерны используем и ПОЧЕМУ
   (каждый выбор обоснован, альтернативы рассмотрены)

3. **Data Flow** — как данные движутся через систему (Mermaid sequence diagram)

4. **Database Changes** — новые таблицы/коллекции, миграции, индексы

5. **Integration Points** — как эта фича взаимодействует с существующим кодом

6. **Risk Assessment** — что может пойти не так + митигация

7. **Non-Functional Requirements** — производительность, масштабируемость, безопасность

Ограничения из constitution.md: [ВСТАВИТЬ КЛЮЧЕВЫЕ ОГРАНИЧЕНИЯ]
```

## ✅ ФАЗА 3: Декомпозиция на задачи

### Промпт 3.1 — Создание tasks.md

```
На основе spec.md и plan.md создай tasks.md — детальный список задач для реализации.

Правила декомпозиции:
- Каждая задача = одна атомарная единица работы (1-4 часа реализации)
- Каждая задача имеет чёткие acceptance criteria
- Указаны зависимости между задачами (что нужно сделать сначала)
- Указан тип задачи: Setup / Schema / Backend / Frontend / Test / Docs

Формат каждой задачи:
```
## Task [N]: [Название]
**Type:** Backend / Frontend / Schema / Test
**Depends on:** Task [X], Task [Y]
**Files affected:** список файлов/модулей

### Description
[Что конкретно нужно сделать]

### Acceptance Criteria
- [ ] [Конкретный проверяемый результат]
- [ ] [...]

### Verification
[Как проверить что задача выполнена правильно]
```

Приоритизируй: инфраструктура и схемы данных → бизнес-логика → API → UI → тесты
```

## 🤖 ФАЗА 4: Имплементация

### Промпт 4.1 — Имплементация задачи (Agent Prompt)

```
Реализуй Task [N] из tasks.md.

## Контекст
- AGENTS.md: [ВСТАВИТЬ СОДЕРЖИМОЕ или путь]
- spec.md: [ВСТАВИТЬ СОДЕРЖИМОЕ или путь]
- plan.md: [ВСТАВИТЬ СОДЕРЖИМОЕ или путь]

## Текущая задача
[ВСТАВИТЬ СОДЕРЖИМОЕ КОНКРЕТНОЙ ЗАДАЧИ ИЗ tasks.md]

## Инструкции
1. Читай AGENTS.md и следуй всем соглашениям — они не обсуждаются
2. Реализуй ТОЛЬКО то, что описано в задаче — ничего лишнего
3. Пиши тесты вместе с кодом (не после)
4. Если что-то непонятно в spec — спроси ПЕРЕД написанием кода
5. После реализации — пройдись по acceptance criteria и подтверди каждый пункт

## Ожидаемый вывод
- Список изменённых/созданных файлов
- Краткое описание принятых решений
- Результат прохождения acceptance criteria
```

### Промпт 4.2 — Верификация (Verifier Agent)

```
Ты — Verifier Agent. Твоя роль: найти проблемы, НЕ подтвердить что всё хорошо.

Проверь реализацию Task [N] против spec.md.

## Что проверять:

**Spec Compliance:**
- [ ] Все outcomes из spec.md реализованы
- [ ] Out-of-scope не добавлен
- [ ] Constraints соблюдены

**Code Quality:**
- [ ] Соответствие conventions из AGENTS.md
- [ ] Нет forbidden patterns из constitution.md
- [ ] Тесты покрывают все acceptance criteria

**Security Check:**
- [ ] Нет hardcoded credentials/secrets
- [ ] Input validation присутствует
- [ ] Аутентификация/авторизация правильная
- [ ] SQL injection / XSS защита

**Edge Cases:**
- [ ] Null/empty input обработан
- [ ] Error responses корректные
- [ ] Граничные значения протестированы

Выведи: список найденных проблем с уровнем severity (BLOCKER / MAJOR / MINOR)
```

## 🔄 ФАЗА 5: Итерация и рефинирование

### Промпт 5.1 — Обновление спецификации после инцидента

```
В production произошёл инцидент: [ОПИСАНИЕ ПРОБЛЕМЫ]

Корневая причина: [ROOT CAUSE]

Обнови spec.md фичи [НАЗВАНИЕ] чтобы:
1. Добавить constraint, который предотвратил бы этот инцидент
2. Добавить verification criteria для edge case который мы пропустили
3. Обновить Out-of-Scope если нужно расширить скоуп

ВАЖНО: Изменения в spec должны автоматически влиять на следующую регенерацию кода.
После обновления spec — укажи какие файлы нужно регенерировать.
```

## 💡 Анти-паттерны SDD (что НЕ делать)

| ❌ Анти-паттерн | ✅ Правильно |
|---|---|
| Vague outcome: "сделать авторизацию" | Конкретный outcome: "JWT + refresh, rate limit 5/15min" |
| Пропустить Out-of-Scope | Явно запретить OAuth: "OAuth is OUT OF SCOPE" |
| Spec после кода | Spec всегда ДО кода |
| Один промпт на всю фичу | Атомарные задачи 1-4 часа |
| Не обновлять spec при инциденте | Инцидент = обновление spec + constraint |
| Доверять AI self-verification | Отдельный Verifier Agent с противоположной целью |
