# Этап 4.1 — Реализация задачи

## Назначение

Реализовать ровно одну утверждённую задачу из `tasks.md`, не расширяя скоуп.

## Примечание о путях

Пути вида `specs/NNN-feature-name/...` в этом prompt — это шаблонный placeholder. Перед работой агент должен использовать реальный feature folder, созданный по принятому naming convention.

## Когда использовать

Используйте только после готовности `spec.md`, `plan.md`, `tasks.md` и прохождения проверки готовности перед реализацией.

## Входные данные

- `specs/NNN-feature-name/spec.md`
- `specs/NNN-feature-name/plan.md`
- `specs/NNN-feature-name/tasks.md`
- `specs/NNN-feature-name/meta.yaml`
- одна выбранная задача для реализации
- релевантные контракты и правила репозитория

## Выходные файлы

```text
Изменённые исходные файлы только для выбранной задачи
```

## Что нужно сделать

1. Сначала проверить `meta.yaml` и убедиться, что feature не находится в статусе `released`, если только это не отдельный подтверждённый change request.
2. Реализовать только одну выбранную задачу из `tasks.md`.
3. Изменять только те файлы, которые действительно нужны для выполнения этой задачи.
4. Сохранять соответствие `spec.md`, `plan.md`, контрактам и правилам репозитория.
5. Не расширять scope задачи даже если рядом видны связанные улучшения.
6. Для UI/user-facing задачи проверить, что изменение достижимо через ожидаемый пользовательский путь или явно зафиксировать Blocker, если текущая задача/декомпозиция не даёт usable workflow.
7. Если в ходе реализации выявлена новая крупная потребность, не реализовывать её сразу, а зафиксировать как follow-up или открытый вопрос.
8. Не изменять поведение released feature без отдельного подтверждения человека.
9. Если отсутствует runnable toolchain или невозможно выполнить runtime smoke для user-facing изменения, не скрывать это как follow-up: отметить как Blocker для release.
10. Не создавать локальный git commit и не выполнять другие операции изменения истории репозитория без явного указания человека. По умолчанию реализация выполняется только через изменения в working tree.

## Критерии готовности

Результат принимается только если:

- реализована ровно одна выбранная задача
- изменения ограничены её scope
- изменения согласованы со `spec.md`, `plan.md` и контрактами
- не появилось молчаливого расширения фичи
- найденные follow-up вопросы зафиксированы явно, а не реализованы скрытно
- для UI/user-facing задачи указано, как изменение попадает в Golden Path, либо зафиксирован Blocker
- локальный git commit не создавался автоматически; если commit требовался, это было явно подтверждено человеком
