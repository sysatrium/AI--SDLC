# Этап 0.1 — Предложение feature slices

## Назначение

Проанализировать project discovery и текущие feature folders в `specs/`, затем предложить следующие feature slices как пользовательские вертикальные срезы.

Этот этап не создаёт новые папки и не создаёт файлы внутри feature folders.

## Discovery path

Используй canonical discovery file:

```text
specs/000-project-overview/discovery.md
```

Не запрашивай discovery у пользователя, если файл существует.
Если файл отсутствует, пустой или заполнен placeholder-значениями вроде `TBA`, явно сообщи, что discovery недостаточен для качественной декомпозиции.

## Что анализировать

- `specs/000-project-overview/discovery.md`
- все numbered feature folders в `specs/`
- demo-фичи исключать из основной декомпозиции
- `000-project-overview` не считать feature

Если у feature есть `meta.yaml`, учитывай её статус.
Если `meta.yaml` нет, считай статус `unknown` и отмечай это явно.

## Что нужно сделать

1. Прочитать discovery.
2. Прочитать существующие numbered features.
3. Определить, какие feature уже `released`, какие ещё активны, а какие имеют неизвестный статус.
4. Найти gaps, overlaps и слишком крупные или не-вертикальные фичи.
5. Предложить candidate feature slices как минимальные пользовательские вертикальные срезы.
6. Для каждой proposed feature объяснить:
   - зачем она нужна;
   - кто основной пользователь;
   - какой end-to-end сценарий она закрывает;
   - что входит в scope;
   - что вне scope;
   - почему это отдельная feature;
   - какие есть вопросы и допущения.
7. Запросить подтверждение по одной следующей feature.
8. Не создавать новые папки и файлы без явного подтверждения.

## Правила декомпозиции

- Feature должна быть пользовательским вертикальным срезом.
- Feature не должна быть просто capability, экраном, API или техническим модулем.
- Feature должна давать самостоятельную пользовательскую или управленческую ценность.
- Feature должна быть достаточно маленькой для отдельного согласования и поставки.
- Feature должна заслуживать отдельный сквозной номер.

## Критерии готовности

Результат принимается только если:

- discovery проанализирован
- существующие features и их статусы учтены
- demo исключены
- candidate slices объяснены через пользовательскую ценность
- открытые вопросы видны явно
- никакие артефакты не созданы
