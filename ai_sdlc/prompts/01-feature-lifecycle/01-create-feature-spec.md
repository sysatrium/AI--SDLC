# Этап 1.1 — Создание feature-спецификации

## Назначение

Создать краткую и проверяемую feature-спецификацию для подтверждённого пользовательского вертикального среза до начала дизайна и кодинга.

## Примечание о путях

- Канонический discovery-файл проекта:

```text
specs/000-project-overview/discovery.md
```

- Пути вида `specs/NNN-feature-name/...` — это шаблонный placeholder.
- Перед работой агент должен использовать реальный feature folder.
- Feature metadata хранится в:

```text
specs/NNN-feature-name/meta.yaml
```

## Когда использовать

Используйте после подтверждения feature на этапе `00-propose-feature-slice` и после создания feature folder.

## Входные данные

- approved proposal из `00-propose-feature-slice`
- `specs/000-project-overview/discovery.md`
- `specs/NNN-feature-name/meta.yaml`
- обзорные артефакты проекта
- релевантные правила конституции
- зависимости и уже принятые решения

## Выходной файл

```text
specs/NNN-feature-name/spec.md
```

## Что нужно сделать

1. Зафиксировать, какую пользовательскую или управленческую задачу решает feature.
2. Сохранить feature как пользовательский вертикальный срез, а не как технический слой или capability.
3. Определить минимальный end-to-end сценарий, который должна закрыть feature.
4. Для user-facing feature явно описать Golden Path: ожидаемый entry point, действие пользователя, успешное завершение сценария и возврат/следующий шаг без знания скрытых URL.
5. Зафиксировать scope, out-of-scope, ограничения, зависимости и уже принятые решения.
6. Явно записать гипотезы, неопределённости и открытые вопросы.
7. Добавить критерии проверки, по которым смогут работать следующие этапы lifecycle.
8. Для user-facing feature добавить критерии достижимости из основного UI, runtime smoke и поведения empty/loading/error states.
9. Не подменять спецификацию деталями реализации.

## Обязательные разделы

- цель
- пользовательский сценарий
- Golden Path и ожидаемый UI/API entry point для user-facing feature
- ожидаемые результаты
- что проверяем / гипотезы
- в скоупе
- вне скоупа
- ограничения
- зависимости
- уже принятые решения
- открытые вопросы
- критерии проверки

## Критерии готовности

Результат принимается только если:

- feature сформулирована как пользовательский вертикальный срез
- результат фичи выражен явно
- границы scope конкретны
- видна связь с discovery и user job
- гипотезы и открытые вопросы записаны явно
- критерии проверки реально проверяемы
- для user-facing feature явно указано, как пользователь достигает и завершает Golden Path
- спецификация не подменена преждевременными деталями реализации
