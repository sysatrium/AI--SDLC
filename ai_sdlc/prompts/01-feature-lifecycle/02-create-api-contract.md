# Этап 1.2 — Создание API-контракта

## Назначение

Определить контракты фичи до реализации, чтобы интерфейсы были явными, минимально достаточными и проверяемыми.

## Примечание о путях

Пути вида `specs/NNN-feature-name/...` в этом prompt — это шаблонный placeholder. Перед работой агент должен использовать реальный feature folder, созданный по принятому naming convention.

## Когда использовать

Используйте после утверждения `spec.md` и до детального планирования реализации или кодинга.

## Входные данные

- `specs/NNN-feature-name/spec.md`
- `specs/NNN-feature-name/meta.yaml`
- релевантные обзорные файлы проекта
- известные требования к интеграциям

## Выходные файлы

```text
specs/NNN-feature-name/contracts/api-spec.yaml
specs/NNN-feature-name/contracts/data-schema.json
specs/NNN-feature-name/contracts/events.yaml
```

## Что нужно сделать

1. Сначала проверить `meta.yaml` и убедиться, что feature не находится в статусе `released`, если только это не отдельный подтверждённый change request.
2. Создавать только те контрактные файлы, которые действительно соответствуют архитектуре и нужны текущему подтверждённому vertical slice.
3. Явно описать запросы, ответы, валидацию и ошибки.
4. Зафиксировать event- или schema-контракты, если фича реально зависит от них.
5. Согласовать контракты со спецификацией фичи, её пользовательским сценарием и границами.
6. Не проектировать универсальные или будущие интерфейсы для соседних неподтверждённых feature.
7. Если контракт нужен только для гипотетического следующего этапа, не создавать его в текущей feature.

## Критерии готовности

Результат принимается только если:

- набор контрактов соответствует реальной архитектуре
- интерфейсы достаточно конкретны для реализации и тестирования
- семантика полей и поведение ошибок описаны явно
- каждый контракт прослеживается к текущему `spec.md`
- контракты не выходят за рамки утверждённого скоупа
- контракты не спроектированы «на вырост» без необходимости текущей feature
