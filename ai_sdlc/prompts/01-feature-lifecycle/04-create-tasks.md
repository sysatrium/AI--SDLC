# Этап 3.1 — Создание задач

## Назначение

Разбить утверждённый план реализации на атомарные и проверяемые задачи, которые агент-реализатор может выполнять по одной.

## Примечание о путях

Пути вида `specs/NNN-feature-name/...` в этом prompt — это шаблонный placeholder. Перед работой агент должен использовать реальный feature folder, созданный по принятому naming convention.

## Когда использовать

Используйте после утверждения `plan.md` и до начала кодинга.

## Входные данные

- `specs/NNN-feature-name/spec.md`
- `specs/NNN-feature-name/plan.md`
- `specs/NNN-feature-name/meta.yaml`
- релевантные контракты

## Выходной файл

```text
specs/NNN-feature-name/tasks.md
```

## Что нужно сделать

1. Разделить работу на атомарные задачи размером обычно не больше 1-4 часов.
2. Для каждой задачи задать тип, зависимости, затрагиваемые файлы, критерии приёмки, способ проверки и ожидаемое evidence.
3. Для каждой задачи указать, какую часть `spec.md` или `plan.md` она реализует.
4. Для каждой задачи обязательно проставить поля `User-facing: yes/no`, `Affects Golden Path: yes/no` и `Expected entry point affected: yes/no`.
5. Явно отмечать задачи, которые являются enablement или технической поддержкой других задач.
6. Сохранить логичный порядок выполнения.
7. Для user-facing feature обязательно добавить задачу или acceptance criteria на app shell/navigation wiring: CTA, ссылки, маршруты, возврат/следующий шаг и достижимость Golden Path из ожидаемого entry point.
8. Не смешивать в одной задаче несколько независимых тем.
9. Не включать в задачи дополнительный scope, который не следует из утверждённой feature.
10. Если задача меняет то, что пользователь видит, нажимает, вводит, куда переходит или как завершает основной сценарий, она должна быть помечена `User-facing: yes`.
11. Если задача может заблокировать, включить или изменить прохождение основного сценария даже как backend, API, validation или integration work, она должна быть помечена `Affects Golden Path: yes`.
12. Если задача меняет точку входа в сценарий, маршрут входа, CTA, меню, стартовый экран или достижимость фичи из ожидаемого UI, она должна быть помечена `Expected entry point affected: yes`.

## Правила

- одна задача — одна ясная единица выполнения
- зависимости должны быть явными
- задачи должны проверяться независимо
- многодневные и расплывчатые задачи запрещены
- если реализация, рефакторинг и дополнительное улучшение UX могут проверяться раздельно, их нельзя сливать в одну задачу
- для user-facing scope запрещено создавать только скрытые routes или технические страницы без пользовательского пути из основного UI
- отсутствие runnable toolchain или среды для проверки фиксируется как Blocker, а не как обычный follow-up
- если задача имеет `User-facing: yes` или `Affects Golden Path: yes`, в ней недостаточно статической проверки кода; нужно указать наблюдаемое runtime evidence или явный Blocker

## Критерии готовности

Результат принимается только если:

- список задач можно выполнять по порядку
- каждая задача атомарна и тестируема
- критерии приёмки конкретны
- проверка возможна по задаче, а не только по фиче целиком
- для user-facing feature есть явная задача или acceptance criteria на достижимость Golden Path
- каждая задача прослеживается к spec или plan
- каждая задача содержит поля `User-facing`, `Affects Golden Path`, `Expected entry point affected` и `Evidence expected`
